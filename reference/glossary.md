# Glossary ŌĆö domain language

The ubiquitous terms for the Finance App. Use these names consistently in code,
schema, specs, and conversation. Maintained alongside the `domain-modeling` skill;
canonical schema detail lives in [../planning/PLAN.md](../planning/PLAN.md).

| Term | Meaning |
|------|---------|
| **Account** | A bank account of a given `type` (`transaction` / `savings` / `loan` / `credit`). Generic by design ŌĆö loans, credit, and cash reserves are *views* over this one model, not separate systems. |
| **Transaction** | One line from a bank CSV: date, amount (negative = money out), raw description, and links to account / merchant / category / import. |
| **Dedupe key** | The natural identity of a transaction: `(account_id, txn_date, amount, description_raw, balance)`. Enforced as a composite `UNIQUE` constraint ŌĆö no hash column. The keystone of idempotent import: a re-import of the same file adds zero rows because all five columns already exist. |
| **ParsedRow** | The intermediate representation of one CSV row, produced by an adapter and consumed by the import engine. Carries: `txn_date`, signed `amount`, `description_raw` (whitespace-stripped bank text ŌĆö feeds the dedupe key), `balance`, `normalised_name_hint` (`str | None` ŌĆö the adapter's merchant name guess; `None` for merchant-less transactions), and `bank_category` (`str | None` ŌĆö bank-supplied category suggestion, carried but ignored in v1). Every adapter produces `ParsedRow` objects; the import engine never reads bank-specific CSV fields directly. |
| **Adapter** | A per-bank module implementing the `BankAdapter` Protocol. Parses one bank's CSV format and returns an `AdapterResult` (a list of `ParsedRow` plus the `reported_balance`). Each adapter owns bank-specific logic: stripping prefixes, converting debit/credit splits to signed amounts, and supplying a `normalised_name_hint` for each row. Returns `None` as the hint for merchant-less rows (internal transfers). One per bank (`bank_a`, `bank_b`); community-extensible ŌĆö adding Bank C is one file + one registry line. |
| **Merchant normalisation** | Mapping a bank's messy raw description to one canonical `normalised_name` in the `merchants` table. Split across two layers: the adapter extracts a `normalised_name_hint` (bank-specific logic ŌĆö prefix stripping, using Bank B's `Details` field, etc.); the import engine title-cases the hint and looks up or creates the `merchants` row. Underpins the new-merchant flag (Phase 5). Categorisation is independent ŌĆö rules run against `description_raw`, not the normalised name. |
| **Merchant** | The normalised payee. No stored `first_seen_at` ŌĆö "first seen" is derived as `MIN(transactions.txn_date)` at flag time, so it stays correct regardless of import order. |
| **Category** | An editable spending category, self-referencing (`parent_id`) for sub-categories. A null category means "Uncategorised". |
| **Merchant-less transaction** | A transaction row with no external merchant to track ŌĆö e.g. an account-to-account transfer, a loan repayment, or payroll income. The adapter signals this by returning `normalised_name_hint = None`; the import engine leaves `merchant_id = NULL`. The new-merchant flag (Phase 5) never fires for these rows. Not all merchant-less transactions are "internal" ŌĆö payroll comes from outside the household but still has no merchant to normalise. |
| **Rule** | An auto-categorisation rule (`contains` / `equals` / `regex` ŌåÆ category). Can be born from clearing an Uncategorised item (the self-improving loop). |
| **Flag** | A transaction marked for review by the flagging engine (over-threshold, double charge, new merchant, ŌĆ”), with a human-readable `reason` and three-state status: `open` / `approved` / `dismissed`. A `double_charge` flag also carries a `related_transaction_id` pointing at the earlier charge. UNIQUE `(transaction_id, flag_type)` prevents duplicate flags on re-runs. |
| **Approve (a flag)** | A family member confirming a flag was worth surfacing ŌĆö sets status to `approved`, records `resolved_by`/`resolved_at`, writes an audit entry. Symmetric with dismiss; no downstream side-effects beyond status + audit. Optionally sets a Merchant Threshold Override for `over_threshold` flags to suppress future alerts on large-but-expected recurring charges. |
| **Dismiss (a flag)** | A family member marking a flag as noise / false positive ŌĆö sets status to `dismissed`. Dismissed flags are the numerator in the false-positive rate metric used to tune thresholds: `dismissed / (approved + dismissed)` per `flag_type`. |
| **Flagging engine** | The service (`services/flagging.py`) that evaluates transactions against three rules ŌĆö `over_threshold`, `double_charge`, `new_merchant` ŌĆö and inserts `Flag` rows. Runs as a separate DB transaction after the import commits. Triggered after every confirmed import (`run_for_import`) and on demand for backfill (`POST /flags/generate`). |
| **Threshold** | The global minimum debit amount (absolute value) that triggers an `over_threshold` flag. Set via `FLAG_THRESHOLD` env var (default `100.00`). Applies to `transaction` and `credit` account types only. Per-merchant overrides take precedence when set. |
| **Merchant Threshold Override** | A per-merchant custom threshold in `merchant_threshold_overrides (merchant_id, threshold, created_by, created_at)`. Takes precedence over the global threshold for `over_threshold` evaluation. Set when approving a recurring large-but-expected charge. |
| **Double-charge window** | The lookback period for `double_charge` detection ŌĆö two transactions with the same `merchant_id` (non-NULL) and same `amount` within N days triggers the flag. N set via `DOUBLE_CHARGE_DAYS` env var (default `7`). |
| **Backfill** | Running the flagging engine against already-imported transactions via `POST /flags/generate`. Idempotent by design ŌĆö the UNIQUE `(transaction_id, flag_type)` constraint with `ON CONFLICT DO NOTHING` makes re-runs safe. |
| **Opening balance** | The immutable anchor on an `Account` ŌĆö the balance immediately before the earliest imported transaction. Set once at account creation, never updated. Used by the reconciliation check. |
| **Current balance** | The cached latest balance on an `Account`, updated each import to the `reported_balance` from that file. Fast for page loads; the reconciliation check detects if it drifts from `opening_balance + ╬Żtxns`. |
| **Reported balance** | The balance the bank states in a CSV file ŌĆö from the preamble for Bank A, from the final row's `Balance` column for Bank B. The import engine writes this to `account.current_balance` after a successful import. Distinct from `opening_balance` (immutable anchor) and `current_balance` (the account's cached value before this import). Importing an older file after a newer one will set `current_balance` backwards; the reconciliation check surfaces the drift. |
| **Audit log** | Chain-of-custody record of every mutating action (imports, reversals, approvals, recategorisations, rule creation). Generic shape: actor, action, target type, target id, optional context. Provenance, **not** access control. |
| **Import engine** | The service (`services/import_engine.py`) that orchestrates the full import pipeline: filename parsing ŌåÆ adapter selection ŌåÆ CSV parsing ŌåÆ merchant normalisation ŌåÆ rule-based categorisation ŌåÆ dedupe insert ŌåÆ balance update ŌåÆ reconciliation check ŌåÆ audit. Consumes `AdapterResult`, writes to the DB, and returns an `ImportResult`. Knows nothing about bank-specific CSV formats ŌĆö all bank logic is in the adapter. One DB transaction: either the whole file commits or nothing does. |
| **Import (batch)** | The structured record of one file's import ŌĆö filename, bank, account, rows added/skipped ŌĆö tying transactions back to where they came from. Append-only; reversal keeps the row and sets `reversed_at`. Distinct from the import engine (the service that runs the import). |
| **Import reversal** | Undoing an import batch: the `imports` row is kept (with `reversed_at`/`reversed_by` set), its transactions are hard-deleted (safe ŌĆö import is idempotent, so re-importable), and an audit entry records the reversal. |
| **Reconciliation** | The check that asserts `opening_balance + ╬Ż(transactions.amount) == current_balance`. `opening_balance` is the immutable anchor (the balance immediately before the earliest imported transaction); `current_balance` is the cached latest value updated each import. Drift means a missing import or dropped row ŌĆö warns loudly. |
| **Lookup table** | A seeded reference table with a natural `text code` primary key (e.g. `account_types`, `flag_types`, `banks`). New values are added with an `INSERT`, no migration needed. FK-referenced from the tables that use them. |
| **Sidecar table** | A 1:1 detail table whose primary key is also a foreign key to the parent (`account_id PK ŌåÆ accounts`). Used for `loan_terms` and `credit_terms` so specialised columns stay in self-describing tables rather than as nullable columns on `accounts`. |
| **B-grade auth** | The day-one security bar: Argon2/bcrypt hashing, real sessions, login rate-limit, MFA hooks present (off until Phase 7). |
| **Budget** | A monthly spending target for a category. Stored in the `budgets` table as `(category_id, valid_from, amount)` ŌĆö one row per category per month it was set. A category with no rows has no budget. |
| **Effective budget** | The budget amount that applies to a given month, determined by rollforward: the most recent `budgets` row where `valid_from Ōēż first day of that month`. |
| **Rollforward** | The carry-forward rule for budgets: the last-set amount applies to every subsequent month until explicitly overridden by a new `budgets` row. Setting a new budget does not affect prior months. |
| **Monthly spend** | The total spending for a category in a calendar month ŌĆö `ABS(SUM(amount WHERE amount < 0))` for transactions in that category (and its sub-categories). Always positive. |
| **Expense category** | A category with net-negative transactions in the current month (money going out). The Spend page shows only expense categories. Categories with net Ōēź 0 (e.g. Income, Transfers) are excluded by the net-positive heuristic. |

> Sign convention (locked): **negative = money out.** Money is always `Decimal` /
> `numeric`, never float.
